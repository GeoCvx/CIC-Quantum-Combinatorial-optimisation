# 当前代码开发结构与进度

## 1. 项目总体定位

当前项目已经形成一套面向 **MILP / MICP** 的混合量子-经典求解框架，核心思路是：

- 离散变量 `x` 由主问题搜索模块负责
- 连续变量 `y` 由子问题精确求解模块负责
- 通过统一评估器把候选 `x` 映射为真实的 `(x, y, Z, r)` 输出

这一总体方向与现有 README 和方案文档中的主线是一致的。README 中已经明确把系统描述为“离散变量启发式搜索 + 连续子问题精确求解”的混合框架；方案文档第 1.2 节也给出了“输入解析 → 主问题代理建模 → 量子候选生成 → 连续子问题真实评估 → 偏置修正 → 局部精修 → 输出结果”的完整闭环流程图。

## 2.2 Master 层

当前已经形成四类主问题搜索模式：

- `classical_master.py`
- `quantum_master.py`
- `hybrid_master.py`
- `iterative_hybrid_master.py`

README 中列出了这四类模式：`classical`、`quantum`、`hybrid`、`iterative_hybrid`；其中 `iterative_hybrid` 已经被定义为当前主力模式。

其中：

### classical_master

负责纯经典搜索，当前已经具备：

- 多起点搜索
- 局部搜索
- 作为 warm start 或对照解使用

### quantum_master

负责量子侧候选生成，当前已经具备：

- QUBO 构造后的量子候选搜索
- QAOA 求解
- 候选解返回

### hybrid_master

负责统一封装 classical / quantum / iterative_hybrid 的调用逻辑，是外部 pipeline 层连接 master 的总入口。

### iterative_hybrid_master

这是当前最完整的主流程实现，已经包含：

- 轮次式迭代
- candidate selection
- subproblem evaluation
- bias / tabu / elite 更新
- local refine
- 迭代停止机制

方案文档第 1.8 和 1.9 节明确给出了“候选筛选—偏置修正—局部搜索”的统一子流程与主流程伪代码；当前 `iterative_hybrid_master` 已经基本对应这一设计框架。

当前进度判断：

- `classical_master`：**已完成**
- `quantum_master`：**已完成**
- `hybrid_master`：**已完成**
- `iterative_hybrid_master`：**已完成并已接入主流程**

## 2.3 Quantum 层

当前已经完成：

- `qubo_builder.py`
- `ising_mapping.py`
- `qaoa_circuit.py`
- `qaoa_solver.py`

README 中已经将这部分明确列为量子模块核心文件。

对应功能如下：

### qubo_builder.py

负责把离散主问题组织成 QUBO 形式。方案文档第 1.4 节与 1.5 节中对“收益项 + 冲突项 + 罚项 + 正则项”的主问题代理建模有完整说明，当前代码已经围绕这一思路完成 QUBO 构造。

### ising_mapping.py

负责 QUBO → Ising 的映射，供量子线路调用。

### qaoa_circuit.py / qaoa_solver.py

负责：

- 构造 QAOA 电路
- 参数搜索
- 采样候选 bitstring
- 返回候选解集合

方案文档第 1.5 节已经把量子模块定位为“候选生成器”而不是直接求原问题全局最优的精确求解器；当前代码与这一定位一致。

当前进度判断：

- **已完成**
- 已能稳定支撑 quantum / hybrid / iterative_hybrid 模式运行

# 4. 当前系统已形成的完整闭环

结合 README 与方案文档，目前代码已经形成的闭环可以总结为：

```
输入 JSON
→ 问题识别（MILP / MICP）
→ 主问题候选生成（classical / quantum / hybrid / iterative_hybrid）
→ candidate selection
→ 连续子问题真实评估
→ 更新 best solution
→ bias / tabu / elite 更新
→ local refine
→ 输出标准答案 JSON
```
